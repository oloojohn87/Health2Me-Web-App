Health2Me Technical Manual

Doctors & Patients

Health2me works by facilitating the communication and networking of doctors and patients in a HIPAA compliant environment. The backbone of the system consists of doctor accounts and patient accounts that are made up of medical records. The system then allows doctors to communicate those records with their patients and with other doctors using a referral system.

Doctors accounts are stored in the doctors table in the database. They can communicate with other doctors, their patients, and also patients that have been referred to them by other doctors. Doctors can also be part of a group of doctors. When a doctor is in a group, he/she can access patient information from other doctors in that group and vice versa. Groups are managed by a set of rules which can be found in “Basic Rules for showing Reports in patientdetailMED”.

Patients accounts are stored in the usuarious table in the database. Patients can communicate with their doctors and doctors that they were referred to. Each patient has a set of medical records all which can be found in the lifepin table in the database.

The information of the patient (also called "Health Passport" has 2 basic components:

  • The Health Summary.
  • The Set of Reports.

The Health Summary is organized and edited by a file called medicalPassport.php.

Medical Records

Medical records for the Health2me system are stored in the lifepin table in the database where each entry represents a medical record. The actual file for the record (pdf, jpg, etc) is encrypted and stored in the Packages_Encrypted folder in the server. The path to the encrypted file for the record is stored in its entry in the lifepin table where it can be obtained and decrypted as needed. More information on how encryption / decryption works can be found in “Health2me Security Encryption Details”.

Medical records have different types. (Demographics, Doctor Notes, Laboratory, Imaging, Patient Notes, Pictures, Superbill, and Summary).

When a doctor refers a patient to another doctor, all of the patient’s medical records are sent to that doctor as well. The sending doctor also has the option to highlight certain records that are important to that referral. Within our system, nothing is actually being sent since all of the records are already in our system. We just grant the receiving doctor’s account access to those records. This is done in the doctorslinkusers table in the database.

Record Privacy

Whenever a patient is referred to another doctor, all of that patient’s files are “sent” to the receiving doctor. However, some records can be marked as private in the database. This is marked as a flag in the IsPrivate column in the lifepin data table. Private records can only be seen by the doctor who created the record. If another doctor that had that patient referred to them wants to see private records, they’ll need to ask the owner of the record for permission.

Messaging

Health2me has an internal messaging system that allows doctors and patients to communicate with each other directly within our system. These messages are stored in the messages_infrasture table in the database for doctor to doctor messages, and in the messages_infrastuctureuser table for doctor to patient and patient to doctor messages. These messages can then be retrieved from the database and displayed to the user as needed. Messages can also have attachments which are just a list of ids of medical records (from the lifepin table) to include with the message. This will allow the receiver of the message to quickly view those medical records once they open the message.

Technologies

The main technologies used in the Health2me system are:

  • HTML / CSS - front-end GUI
  • Javascript (with JQuery) - front end processing
  • PHP - back end processing
  • MYSQL - database for storage of system data
  • Apache - application server

Code Structure

Basic Processing Diagram

Front End

In Health2me, front-end pages are loaded dynamically with php first and then displayed as html / css. We can then manipulate the page further using Javascript and JQuery for front-end processing.

  1. First, the basic information for the particular user, as well as other important information for the page, is obtained and stored with php before the page is loaded. This usually happens at the very top of the code.
  2. Second, the actual GUI of the page is loaded with html / css. Some dynamic information can also be inserted here with embedded php calls. i.e. <input type=“text” value=“<?php echo $username; ?>” > inserts the current user’s username as the value for this input text field.
  3. Finally, we have the javascript / jquery code at the bottom of the page which does all of the front-end processing. Here we can change the page by dynamically changing the page’s attributes and make asynchronous calls to the server to obtain data.

Back End

The back-end of the system is handled by PHP with a connection to our mysql database. Data can be queried from the database and also updated/inserted into the database from PHP. This data can then be encoded and stringified in to a JSON (Javascript Object Notation) object and then sent back to the front-end as needed.
